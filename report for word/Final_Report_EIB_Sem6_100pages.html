1.2 Problem Statement

Mental health challenges affect millions of people worldwide, with stress, anxiety, and mood disorders becoming increasingly prevalent in modern society. However, traditional approaches to mental health management often lack the following:

  1. Accessibility: Professional mental health services remain inaccessible to many due to cost, stigma, or geographical limitations.
  2. Continuous Monitoring: Traditional therapy sessions provide only periodic check-ins, missing the day-to-day fluctuations in emotional states that could provide valuable insights.
  3. Data-Driven Insights: Without systematic tracking, patterns and triggers of mood changes remain unidentified, making it difficult to implement effective interventions.
  4. Personal Empowerment: Many individuals lack tools to take an active role in understanding and managing their emotional health.
  5. Privacy Concerns: Discussing mental health issues often raises privacy concerns, deterring many from seeking help or tracking their emotional states.
  6. Contextual Understanding: Traditional mood tracking methods often fail to capture the contextual factors influencing emotional states, limiting the depth of insights that can be gained.

The Pro Mood Tracker addresses these challenges by providing a private, accessible, and comprehensive tool for continuous mood monitoring and analysis, empowering users to take control of their emotional well-being through data-driven insights.

2. Background and Literature Review

2.1 Existing Mood Tracking Solutions

The landscape of mood tracking applications has evolved significantly over the past decade, with various solutions offering different approaches to emotional monitoring and analysis. A comprehensive review of existing solutions reveals several categories:

Commercial Applications

Commercial mood tracking applications such as Daylio, Moodpath, and MoodMission have gained popularity for their user-friendly interfaces and gamified approaches to mood tracking. These applications typically offer basic mood logging functionality with limited analysis capabilities in their free versions, while reserving more advanced features for premium subscriptions.

Key features commonly found in commercial applications include:

  • Simple mood scales (often emoji-based)
  • Activity correlation
  • Basic statistics and trends
  • Reminder systems
  • Journal entries

Academic and Therapeutic Tools

In contrast to commercial applications, mood tracking tools developed for academic research or therapeutic purposes tend to prioritize data quality and validated assessment methods over user engagement. Examples include the PANAS (Positive and Negative Affect Schedule) mobile implementations and MoodPrism, which incorporate standardized psychological assessments.

These applications generally feature:

  • Validated psychological scales
  • More detailed mood assessment
  • Research-oriented data collection
  • Limited customization options
  • Less emphasis on visual design and user experience
6

Gaps in Existing Solutions

While the market offers numerous mood tracking options, several notable gaps persist:

  1. Contextual Intelligence: Most applications collect limited contextual data or fail to meaningfully correlate this data with mood states.
  2. Insight Generation: Many applications present data visualizations but offer minimal actionable insights based on the collected information.
  3. Privacy-Focused Design: Few applications prioritize local data processing and user control over information sharing.
  4. Holistic Integration: Limited integration with other health and wellness data sources restricts the comprehensive understanding of factors affecting emotional well-being.
  5. Balance of Engagement and Scientific Validity: Applications tend to favor either user engagement or scientific rigor, rarely achieving a balanced approach.

The Pro Mood Tracker aims to address these gaps by combining engaging user experience with scientifically-informed approaches to mood assessment, contextual data collection, and insight generation while maintaining strong privacy protections.

2.2 Psychological Foundations

The Pro Mood Tracker's design is grounded in several established psychological principles and frameworks related to emotional well-being and self-monitoring:

Dimensional Models of Emotion

Rather than treating emotions as discrete categories, the application adopts Russell's Circumplex Model of Affect, which conceptualizes emotions along two primary dimensions: valence (positive to negative) and arousal (high to low energy). This dimensional approach allows for more nuanced emotion tracking compared to basic "good/bad" scales found in many applications.

Cognitive Behavioral Theory

The application incorporates principles from Cognitive Behavioral Therapy (CBT), particularly the connection between thoughts, emotions, and behaviors. By prompting users to record not only their mood but also associated activities and thoughts, the application helps users identify patterns consistent with CBT's emphasis on recognizing cognitive and behavioral factors affecting emotional states.

7

Self-Determination Theory

Self-Determination Theory (SDT) informs the engagement features of the Pro Mood Tracker, focusing on satisfying the psychological needs of autonomy, competence, and relatedness. The application's approach to customization, achievement systems, and optional social features aligns with SDT's principles for fostering intrinsic motivation.

Ecological Momentary Assessment

The methodology of Ecological Momentary Assessment (EMA), which involves repeated sampling of subjects' behaviors and experiences in real time in their natural environments, underpins the Pro Mood Tracker's approach to data collection. This method reduces recall bias and provides more accurate insights into emotional patterns compared to retrospective reporting.

Positive Psychology

Concepts from Positive Psychology, including the emphasis on cultivating positive emotions and identifying personal strengths, influence the application's focus on not only tracking problems but also highlighting positive experiences and patterns that contribute to well-being.

3.2 Data Dictionary

The Pro Mood Tracker application manages several key data entities. The following data dictionary defines these entities and their attributes:

User

  • userId: String (Primary Key) - Unique identifier for each user
  • email: String - User's email address for authentication
  • displayName: String - User's chosen display name
  • createdAt: Timestamp - When the user account was created
  • lastLoginAt: Timestamp - When the user last logged in
  • preferences: Object - User's application preferences

MoodEntry

  • entryId: String (Primary Key) - Unique identifier for each mood entry
  • userId: String (Foreign Key) - Reference to the user who created the entry
  • timestamp: Timestamp - When the mood was recorded
  • moodScore: Number (1-10) - Numerical rating of mood intensity
  • moodLabel: String - Descriptive label for the mood (e.g., "Happy", "Anxious")
  • notes: String - Optional user notes about the mood entry
  • location: GeoPoint - Optional location data when mood was recorded
  • isModified: Boolean - Indicates if entry was edited after creation

ContextFactor

  • factorId: String (Primary Key) - Unique identifier for each context factor
  • entryId: String (Foreign Key) - Reference to associated mood entry
  • factorType: String - Type of factor (e.g., "Activity", "Sleep", "Weather")
  • factorValue: Mixed - Value of the factor (type-dependent)
  • intensity: Number (1-5) - Optional intensity rating for the factor

ActivityLog

  • activityId: String (Primary Key) - Unique identifier for each activity
  • userId: String (Foreign Key) - Reference to the user who performed the activity
  • activityType: String - Type of activity (e.g., "Exercise", "Social", "Work")
  • startTime: Timestamp - When the activity started
  • duration: Number - Duration in minutes
  • intensity: Number (1-5) - Subjective intensity of the activity
  • notes: String - Optional user notes about the activity

Insight

  • insightId: String (Primary Key) - Unique identifier for each insight
  • userId: String (Foreign Key) - Reference to the user receiving the insight
  • generatedAt: Timestamp - When the insight was generated
  • insightType: String - Type of insight (e.g., "Correlation", "Pattern", "Trend")
  • description: String - Human-readable description of the insight
  • confidence: Number (0-1) - Confidence level in the insight's validity
  • relatedFactors: Array - Factors related to this insight
  • isRead: Boolean - Whether the user has viewed this insight

This data structure enables comprehensive mood tracking while maintaining flexibility for user customization and extension for future features.

3.4 Database Design

The Pro Mood Tracker implements a hybrid database architecture combining local SQLite storage with Cloud Firestore for synchronization. This design supports the offline-first approach while enabling cross-device synchronization when connectivity is available.

Local Storage (SQLite)

The local database uses SQLite with the following table structure:

CREATE TABLE users (
    userId TEXT PRIMARY KEY,
    email TEXT UNIQUE,
    displayName TEXT,
    createdAt INTEGER,
    lastLoginAt INTEGER,
    preferences TEXT
);

CREATE TABLE mood_entries (
    entryId TEXT PRIMARY KEY,
    userId TEXT,
    timestamp INTEGER,
    moodScore REAL,
    moodLabel TEXT,
    notes TEXT,
    location TEXT,
    isModified INTEGER,
    syncStatus TEXT,
    FOREIGN KEY (userId) REFERENCES users(userId)
);

CREATE TABLE context_factors (
    factorId TEXT PRIMARY KEY,
    entryId TEXT,
    factorType TEXT,
    factorValue TEXT,
    intensity REAL,
    syncStatus TEXT,
    FOREIGN KEY (entryId) REFERENCES mood_entries(entryId)
);

CREATE TABLE activity_logs (
    activityId TEXT PRIMARY KEY,
    userId TEXT,
    activityType TEXT,
    startTime INTEGER,
    duration REAL,
    intensity REAL,
    notes TEXT,
    syncStatus TEXT,
    FOREIGN KEY (userId) REFERENCES users(userId)
);

CREATE TABLE insights (
    insightId TEXT PRIMARY KEY,
    userId TEXT,
    generatedAt INTEGER,
    insightType TEXT,
    description TEXT,
    confidence REAL,
    relatedFactors TEXT,
    isRead INTEGER,
    syncStatus TEXT,
    FOREIGN KEY (userId) REFERENCES users(userId)
);

Cloud Storage (Firestore)

The Firestore database uses a document-oriented structure with the following collections:

  • users/ - User profiles and authentication information
  • users/{userId}/moodEntries/ - Mood entry documents for a specific user
  • users/{userId}/contextFactors/ - Context factors associated with mood entries
  • users/{userId}/activityLogs/ - Activity logging information
  • users/{userId}/insights/ - Generated insights for a specific user
  • users/{userId}/settings/ - User-specific application settings

Synchronization Strategy

The synchronization between local and cloud storage follows these principles:

  1. Offline-First: All data operations first occur in the local database
  2. Change Tracking: Modified records are marked with syncStatus flags
  3. Conflict Resolution: Timestamp-based resolution with user notification for conflicts
  4. Batched Sync: Synchronization occurs in batches to minimize bandwidth usage
  5. Selective Sync: Users can specify which data types to synchronize

Data Security Measures

To ensure user privacy and data security, the following measures are implemented:

  • Local database encryption using SQLCipher
  • Cloud data encrypted at rest and in transit
  • Firestore security rules enforcing per-user data isolation
  • Optional end-to-end encryption for sensitive fields
  • User-controlled data retention policies
  • Automated data backup and recovery mechanisms

This hybrid database design balances the need for responsive local operations with the benefits of cloud synchronization, while maintaining strong privacy and security protection for sensitive user mood data.

Mood Correlation Algorithm

The following pseudocode illustrates how the Pro Mood Tracker identifies correlations between contextual factors and mood ratings:

FUNCTION CalculateMoodCorrelations(userId, timeframe):
    // Retrieve mood entries for the specified user and timeframe
    moodEntries = Database.GetMoodEntries(userId, timeframe)
    
    // Initialize map to store factor correlations
    factorCorrelations = {}
    
    // Group entries by factor types (activities, weather, etc.)
    FOR EACH factorType IN GetFactorTypes():
        // Extract mood ratings and factor values
        moodValues = []
        factorValues = []
        
        FOR EACH entry IN moodEntries:
            IF entry.HasFactor(factorType):
                moodValues.Add(entry.moodRating)
                factorValues.Add(entry.GetFactorValue(factorType))
        
        // Calculate Pearson correlation coefficient if enough data points
        IF moodValues.Length >= 5:
            correlation = CalculatePearsonCorrelation(moodValues, factorValues)
            significance = CalculateStatisticalSignificance(correlation, moodValues.Length)
            
            // Store if correlation is statistically significant
            IF significance < 0.05:
                factorCorrelations[factorType] = {
                    correlation: correlation,
                    significance: significance,
                    dataPoints: moodValues.Length
                }
    
    // Sort correlations by absolute strength
    sortedCorrelations = SortByAbsoluteValue(factorCorrelations)
    
    // Return top correlations with descriptive labels
    FOR EACH corr IN sortedCorrelations:
        AddDescriptiveLabel(corr)
    
    RETURN sortedCorrelations

Streak Maintenance Algorithm

The following pseudocode demonstrates how the application tracks and maintains user streaks for engagement:

FUNCTION UpdateUserStreak(userId):
    // Get user info and last mood entry
    user = Database.GetUser(userId)
    lastEntry = Database.GetLastMoodEntry(userId)
    currentDate = GetCurrentDate()
    
    // If first entry, initialize streak
    IF lastEntry IS NULL:
        user.streakDays = 1
        user.lastStreakDate = currentDate
        Database.UpdateUser(user)
        RETURN 1
    
    // Calculate days between last entry and today
    daysSinceLastEntry = CalculateDaysBetween(lastEntry.date, currentDate)
    
    // Update streak based on timing
    IF daysSinceLastEntry == 0 AND lastEntry.date == currentDate:
        // Already logged today, streak unchanged
        RETURN user.streakDays
    ELSE IF daysSinceLastEntry == 1 OR 
           (daysSinceLastEntry == 0 AND lastEntry.date == Yesterday()):
        // Consecutive day entry, increment streak
        user.streakDays += 1
        user.lastStreakDate = currentDate
    ELSE IF daysSinceLastEntry > 1:
        // Streak broken, reset to 1
        user.streakDays = 1
        user.lastStreakDate = currentDate
    
    // Check for streak-based achievements
    CheckStreakAchievements(userId, user.streakDays)
    
    // Update user record
    Database.UpdateUser(user)
    
    RETURN user.streakDays
